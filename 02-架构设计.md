# 第二章：架构设计

> 架构就是"分工"。谁做什么，怎么沟通。

---

## 2.1 架构模式：前后端分离

### 为什么要分离？

**❌ 不分离的写法（远古时代）**：
```php
<!-- index.php -->
<html>
<body>
<?php
// 这里直接查数据库
$user = $db->query("SELECT * FROM users WHERE id = 1");
echo "<h1>欢迎 " . $user['name'] . "</h1>";
?>
</body>
</html>
```
- HTML、PHP、SQL 混在一起
- 改样式要改 PHP，改逻辑要碰 HTML
- 前端工程师不会 PHP，后端不会 CSS，没法分工

**✅ 前后端分离**：

```
┌─────────────┐      HTTP 请求      ┌─────────────┐
│   前端     │  ←───────────────  │   后端     │
│ (浏览器/APP)│  ───────────────→  │ API 服务   │
│            │      JSON 数据      │            │
└─────────────┘                     └─────────────┘
```

- **前端**：负责页面展示、用户交互
- **后端**：负责数据存储、业务逻辑、API 接口
- **分工明确**：前端管怎么显示，后端管怎么存数据

### 前后端职责划分

| 职责 | 前端 | 后端 |
|------|------|------|
| 页面展示 | ✅ | - |
| 用户交互 | ✅ | - |
| 路由控制 | ✅（SPA） | ✅（传统） |
| 表单验证 | ✅（体验） | ✅（安全） |
| 业务逻辑 | ✅（简单的） | ✅（核心） |
| 数据存储 | - | ✅ |
| 安全认证 | - | ✅ |
| API 接口 | - | ✅ |

### 常见架构模式对比

#### 1. 前后端分离（推荐现代项目）
- ✅ 分工清晰，好协作
- ✅ 前端可以做 APP、小程序多端复用
- ✅ 前后端可以独立开发、独立部署
- ❌ 需要跨域处理，需要 API 文档

#### 2. 服务端渲染（SSR）
- 后端生成 HTML 发给浏览器
- 优点：SEO 好，首屏快
- 缺点：前后端还是混在一起，协作麻烦

#### 3. 全栈在前端（JAMstack / Vercel）
- 前端 + Serverless
- 小项目可以，大项目还是需要后端

## 2.2 客户端 / 服务端分工

### 基本概念

- **客户端（Client）**：用户直接用的那个东西
  - 浏览器打开的网页
  - 手机上的 APP
  - 你的电脑上的桌面程序

- **服务端（Server）**：在后台跑，给客户端提供服务
  - 存数据
  - 算东西
  - 处理请求

- **请求响应**：
  ```
  客户端 → 请求（我要xxx） → 服务端
  客户端 ← 响应（给你xxx） ← 服务端
  ```

### 分工原则： heavier on the server？还是 heavier on client？

#### 方案 A：逻辑放后端
```python
# 后端：算出结果给客户端
@app.get("/statistics")
def get_stats():
    # 复杂计算在后端做
    data = calculate_complex_stats()
    return {"result": data}
```
- ✅ 数据安全，不容易被篡改
- ✅ 客户端轻，弱设备也能用
- ❌ 每次操作都要请求，慢一点

#### 方案 B：逻辑放客户端
```javascript
// 前端：拿到数据自己算
fetch('/all-data')
  .then(res => res.json())
  .then(data => {
    // 复杂计算在前端做
    const result = calculateComplexStats(data)
    render(result)
  })
```
- ✅ 用户操作快，不用等网络
- ❌ 数据都给客户端了，不安全
- ❌ 客户端耗电快

### 📌 分工建议

| 类型 | 放哪里 | 原因 |
|------|--------|------|
| 展示、交互 | 客户端 | 用户操作要快 |
| 数据存储 | 服务端 | 持久化，安全 |
| 复杂计算 | 服务端 | 客户端算不动 |
| 验证安全 | 服务端 | 客户端不可信 |
| 简单过滤排序 | 客户端 | 减少请求 |

> 记住：**客户端不可信**！任何安全检查一定要在后端再做一遍。

## 2.3 通信模式：API 接口

### 什么是 API 接口？

> API 就是后端给前端留的"出入口"。

你想做什么事，就调用对应的 API：

| 你想做什么 | 调用哪个 API |
|-----------|-------------|
| 获取用户信息 | `GET /api/user/123` |
| 创建新待办 | `POST /api/todos` |
| 更新待办 | `PUT /api/todos/123` |
| 删除待办 | `DELETE /api/todos/123` |

### HTTP 方法对应操作

| 方法 | 用途 | 例子 |
|------|------|------|
| **GET** | 获取数据 | 查用户信息、查列表 |
| **POST** | 创建数据 | 新增、提交表单 |
| **PUT/PATCH** | 更新数据 | 修改信息 |
| **DELETE** | 删除数据 | 删除 |

### URL 设计

**❌ 不好的设计**：
```
/getUserById
/getAllTodos
/addTodo
/updateTodoContent
/deleteTodo
```
- 动词放 URL 里，重复
- 看不出来资源是什么

**✅ RESTful 设计**：
```
# 资源是 todos
GET    /api/todos       → 获取列表
GET    /api/todos/{id}  → 获取单个
POST   /api/todos       → 新建
PUT    /api/todos/{id}  → 更新整体
PATCH  /api/todos/{id}  → 更新部分
DELETE /api/todos/{id}  → 删除
```

- **URL 定位资源**：用名词
- **方法表示操作**：GET/POST/PUT/DELETE

### 请求响应流程

**完整例子：用户新增待办**

```
 1. 用户输入内容，点击"保存"
 │
 2. 前端：
 │   - 验证内容不为空
 │   - 发请求：POST /api/todos
 │   - Body: {"content": "买牛奶"}
 ↓
 3. 网络传输到后端
 ↓
 4. 后端：
 │   - 验证用户登录（Token）
 │   - 验证数据格式
 │   - 插入数据库
 │   - 返回：{"id": 123, "content": "买牛奶", "status": "created"}
 ↓
 5. 网络传输回前端
 ↓
 6. 前端：
 │   - 收到结果
 │   - 把新待办加到列表
 │   - 刷新页面显示
 └── 完成 ✅
```

### 请求响应格式

**请求**一般是：
```json
{
  "data": "xxx",
  "page": 1,
  "size": 10
}
```

**响应**一般统一格式：
```json
{
  "code": 0,          // 0 表示成功，非 0 表示错误
  "message": "ok",    // 错误信息
  "data": {           // 实际数据
    "id": 123,
    "content": "买牛奶"
  }
}
```

**错误响应**：
```json
{
  "code": 1001,
  "message": "内容不能为空",
  "data": null
}
```

统一格式好处：前端可以统一处理，不用每个请求写不同判断。

## 2.4 RESTful 规范：让 API 更清晰

### 核心原则

1. **URL 用名词，不用动词**
   - ✅ `/users` 不是 `/getUsers`
   - ✅ `/todos/123` 不是 `/getTodo?id=123`

2. **用 HTTP 方法表示操作**
   - GET = 读
   - POST = 创
   - PUT = 改
   - DELETE = 删

3. **分层 URL**
   ```
   /users/{userId}/todos
   → 获取某个用户的所有待办
   ```
   关系清晰，一看就懂

4. **用 HTTP 状态码**

| 状态码 | 意思 | 什么时候用 |
|--------|------|-----------|
| 200 | OK | 请求成功 |
| 201 | Created | 创建成功 |
| 400 | Bad Request | 参数错了 |
| 401 | Unauthorized | 没登录 |
| 403 | Forbidden | 登录了，但没权限 |
| 404 | Not Found | 资源不存在 |
| 500 | Internal Server Error | 服务器错了 |

### 常见误区

**❌ 误区 1：只用 GET 和 POST**
```
/getAllTodos → ❌ 应该是 GET /todos
/addTodo     → ❌ 应该是 POST /todos
/updateTodo  → ❌ 应该是 PUT /todos/{id}
/deleteTodo  → ❌ 应该是 DELETE /todos/{id}
```

**❌ 误区 2：URL 嵌套太深**
```
/users/{userId}/todos/{todoId}/comments/{commentId}
```
- 太深了，不好维护
- 一般 2-3 层就够了

**❌ 误区 3：过度追求 RESTful**
- 不是所有东西都要严格 RESTful
- 一些操作不对应 CRUD，用 POST 没关系
  ```
  POST /auth/login  → 完全可以，不用纠结
  ```

### 完整示例：待办 API 设计

```
# 待办管理
GET    /api/todos          → 获取待办列表
GET    /api/todos?status=completed  → 过滤已完成
POST   /api/todos          → 新建待办
GET    /api/todos/{id}     → 获取单个待办
PUT    /api/todos/{id}     → 更新待办
DELETE /api/todos/{id}     → 删除待办
POST   /api/todos/{id}/complete  → 标记完成（非 CRUD，没问题）

# 用户管理
POST   /api/auth/login     → 登录
GET    /api/auth/me        → 获取当前用户信息
DELETE /api/auth/logout    → 登出
```

## 2.5 状态管理

### 什么是状态？

> "状态"就是"当前的数据"。

- 前端状态：当前用户是谁，待办列表是什么，哪些选中了
- 后端状态：数据库里存的数据，就是持久化状态

### 前端状态管理

#### ❌ 不好：到处乱存

```javascript
// 每个组件自己发请求，自己存
// 组件 A 改了数据，组件 B 不知道，数据不同步
let todos = []
function ComponentA() {
  fetch('/todos').then(res => {
    todos = res.json()
  })
}
```

#### ✅ 好：集中管理

**方案 1：简单项目 - 放顶层组件**
```javascript
// App.jsx 顶层组件统一存
const [todos, setTodos] = useState([])
// 所有子组件通过 props 拿到
<TodoList todos={todos} />
<AddTodo onAdd={setTodos} />
```

**方案 2：复杂项目 - 用状态管理库**
- React: Redux / Zustand / Jotai
- Vue: Vuex / Pinia

好处：
- 数据只有一份
- 改了之后，所有用到的地方自动更新
- 数据同步不会错

### 后端业务逻辑分层

前面第一章讲了分层，再强调一下：

```
┌─────────────────────────────────────────────────┐
│                   API 层（Controller）            │
│  职责：处理 HTTP 请求，调用 Service，返回响应    │
├─────────────────────────────────────────────────┤
│                业务层（Service）                  │
│  职责：处理业务逻辑，调用 Repository             │
├─────────────────────────────────────────────────┤
│              数据访问层（Repository）             │
│  职责：只做数据库操作                            │
└─────────────────────────────────────────────────┘
```

### 每层该做什么，不该做什么

**❌ 错误：越层**
```python
# API 层直接操作数据库，跳过 Service
@app.post("/todos")
def add_todo():
    # API 直接写 SQL，业务逻辑没地方放了
    db.execute("INSERT ...")
    return {}
```

**✅ 正确：每层只做自己的事**
```python
# api/todos.py → 只处理请求
from services.todo_service import TodoService
@router.post("/todos")
def add_todo(request):
    return TodoService().add(request.content)

# services/todo_service.py → 只处理业务逻辑
from repositories.todo_repo import TodoRepository
class TodoService:
    def add(self, content):
        # 验证
        if not content:
            raise ValueError("空内容")
        # 调用仓储保存
        return TodoRepository().insert(content)

# repositories/todo_repo.py → 只操作数据库
class TodoRepository:
    def insert(self, content):
        sql = "INSERT INTO todos (content) VALUES (?)"
        db.execute(sql, (content,))
        return db.last_insert_id()
```

### 为什么要分层？

- **改 API 不影响业务**：换 Web 框架，只改 API 层
- **改存储不影响业务**：换数据库，只改 Repository 层
- **可测试**：可以单独测试 Service 层业务逻辑
- **分工**：不同人做不同层，不打架

## 第二章 总结

### 核心要点

| 要点 | 一句话总结 |
|------|-----------|
| **前后端分离** | 前端管展示，后端管数据，分工协作 |
| **C/S 分工** | 客户端做交互，服务端做存储和安全 |
| **API 接口** | 前后端通过 JSON 请求响应通信 |
| **RESTful** | URL 用名词，方法表操作，清晰规范 |
| **状态管理** | 前端集中管理，后端分层清晰 |
| **业务分层** | API → Service → Repository，职责分明 |

### 避开 vibecoding 陷阱

❌ API、业务、SQL 混在一个函数
❌ URL 设计混乱，到处是动词
❌ 前端状态到处乱存，数据不同步
❌ 安全验证只做前端，不做后端
❌ 前后端不分离，混写 PHP/HTML

✅ 分工清晰，分层明确
✅ URL 设计符合 RESTful
✅ 前端状态集中管理
✅ 安全验证一定在后端重做
✅ 请求响应统一格式，方便处理
