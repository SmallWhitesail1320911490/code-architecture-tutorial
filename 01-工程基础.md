## 1.1 需求拆解：把大象装进冰箱分几步

### 核心思想
**大问题拆成小问题，逐个解决。**

很多人看到复杂需求就头晕，其实只要会拆解，再复杂的问题也能搞定。

### 练习示例：做一个待办事项 App

**❌ 错误拆解（vibecoding 常见）**：
```
直接开始写代码：
- 打开编辑器
- 写个 main 函数
- 想到什么写什么
- 跑起来有问题再改
```

**✅ 正确拆解**：

```
待办事项 App
├── 1. 新增待办
│   ├── 用户输入内容
│   ├── 验证内容不为空
│   └── 保存到数据库
├── 2. 查看待办列表
│   ├── 从数据库读取所有待办
│   ├── 按创建时间排序
│   └── 显示出来
├── 3. 完成待办
│   ├── 用户点击"完成"按钮
│   ├── 更新状态为已完成
│   └── 保存更改
└── 4. 删除待办
    ├── 用户点击删除
    ├── 确认提醒
    └── 从数据库删除
```

### 📝 拆解原则

1. **MECE 原则**：相互独立，完全穷尽
   - 每个子问题不重叠
   - 不遗漏任何功能

2. **直到能直接写代码**：
   - 不能再拆了，就开始写
   - 如果一个函数超过 50 行，说明还能拆

3. **从上到下，逐步细化**：
   - 先粗分，再细分
   - 不要一开始就陷入细节

### 常见错误

| 错误 | 为什么不好 |
|------|-----------|
| 一次性想写完所有功能 | 顾头不顾尾，到处是 bug |
| 不拆解直接写 | 写到一半发现逻辑混乱 |
| 拆解太细，每个函数只有一行 | 过度设计，读起来费劲 |

## 1.2 复杂度管理：别让代码炸掉

### 核心思想
**控制每个模块的复杂度，不让它"爆炸"。**

### 怎么管理复杂度

#### 1. 拆分文件，不要千行大文件

**❌ 错误写法：**
```python
# main.py 1000 行
# 所有东西都在这里面
app = FastAPI()
model = load_model()
# ... 1000 行代码 ...
```

**✅ 正确写法：**
```
my_app/
├── api/          # API 接口
├── services/     # 业务逻辑
├── models/       # 数据模型
└── core/         # 工具函数
```

#### 2. 控制函数长度

- ✅ 一个函数只做一件事
- ✅ 一般控制在 30 行以内
- ✅ 超过 50 行就该拆分了

**❌ 错误：一个函数做十件事**
```python
def process_order():
    check_user()      # 检查用户
    check_stock()     # 检查库存
    calculate_price() # 计算价格
    charge_money()    # 扣款
    send_notification() # 发通知
    update_inventory()  # 更新库存
    # ... 太多了 ...
```

**✅ 正确：拆分后清晰多了**
```python
def process_order():
    check_user()
    check_stock()
    calculate_price()
    charge_money()
    send_notification()
    update_inventory()

# 虽然行数差不多，但每个步骤都清晰
# 要改哪里就去哪，不会乱
```

#### 3. 减少嵌套层次

**❌ 地狱嵌套，谁看谁晕**
```python
if user:
    if user.login:
        if order:
            if order.status == 'pending':
                process()
            else:
                print('status error')
        else:
            print('order not found')
    else:
        print('not login')
else:
    print('user not found')
```

**✅ 提前返回，扁平化**
```python
if not user:
    print('user not found')
    return

if not user.login:
    print('not login')
    return

if not order:
    print('order not found')
    return

if order.status != 'pending':
    print('status error')
    return

process()  # 主逻辑放最后，一眼看到
```

> 黄金法则：嵌套不要超过 3 层

## 1.3 面向对象编程（OOP）：把数据和动作放一起

### 什么是面向对象？

**通俗理解：把"东西"和"它能做的事"打包在一起。**

比如：
- **人**：有姓名、年龄 → 能吃饭、能走路
- **汽车**：有品牌、颜色 → 能启动、能加速
- **用户**：有用户名、邮箱 → 能登录、能修改资料

### 基础示例

**❌ 面向过程写法（数据和行为分离）**
```python
# 数据分开放
user_name = "zhangsan"
user_email = "zhangsan@example.com"
user_password = "hashed_password"

# 函数单独写
def login(name, password):
    # 验证密码
    pass

def change_email(name, new_email):
    # 修改邮箱
    pass

# 用的时候要把数据传进来
login(user_name, user_password)
```

**✅ 面向对象写法（打包在一起）**
```python
class User:
    def __init__(self, name, email, password):
        self.name = name
        self.email = email
        self.password_hash = password

    def login(self, password):
        # self 就是当前用户自己
        return self.verify_password(password)

    def change_email(self, new_email):
        # 直接用 self.email，不用传
        self.email = new_email

# 创建用户
user = User("zhangsan", "zhangsan@example.com", "hashed_password")

# 用起来很直观
user.login("password123")
user.change_email("new@example.com")
```

### 三大特性

#### 1. 封装：把细节藏起来

只暴露必要的接口，内部细节不用让别人知道。

```python
class BankAccount:
    def __init__(self):
        self.__balance = 0  # 私有，外面不能直接改

    def deposit(self, amount):
        # 存款：会验证金额
        if amount > 0:
            self.__balance += amount

    def withdraw(self, amount):
        # 取款：会检查余额
        if amount <= self.__balance:
            self.__balance -= amount
            return amount
        return 0

    def get_balance(self):
        return self.__balance

account = BankAccount()
account.deposit(1000)      # ✅ 只能通过方法操作
account.withdraw(500)      # ✅ 有验证，不会出错
# account.__balance = 1000000 ❌ 外面不能直接改，防止乱改
```

#### 2. 继承：复用代码

```python
# 基类：通用功能
class Animal:
    def __init__(self, name):
        self.name = name

    def eat(self):
        print(f"{self.name} is eating")

# 子类：继承后自动拥有父类的方法
class Dog(Animal):
    def bark(self):
        print(f"{self.name} is barking")

class Cat(Animal):
    def meow(self):
        print(f"{self.name} is meowing")

dog = Dog("旺财")
dog.eat()    # ✅ 继承来的，不用再写一遍
dog.bark()   # ✅ 自己特有的方法
```

#### 3. 多态：不同东西，同样用法

```python
class Dog(Animal):
    def speak(self):
        return "汪汪"

class Cat(Animal):
    def speak(self):
        return "喵喵"

# 不管是狗还是猫，都能 speak
animals = [Dog("旺财"), Cat("咪咪")]
for animal in animals:
    print(animal.speak())
# 输出：
# 汪汪
# 喵喵
```

### 什么时候用 OOP？

✅ **推荐用**：
- 需要管理**状态**（有变化的数据）
- 需要创建多个相似对象
- 数据和操作紧密相关

❌ **不用过度用**：
- 简单脚本
- 只是一次计算
- 没有状态变化

## 1.4 模块化：高内聚，低耦合

### 什么是模块化？

> **把相关的东西放一起，和外界关系理清。**

- **高内聚**：同一个模块内的东西关系紧密
- **低耦合**：模块之间尽量少互相依赖

### 模块化示例：待办应用

**❌ 混乱版本：所有东西混在一起**
```python
# app.py
from fastapi import FastAPI
import sqlite3

app = FastAPI()

@app.post("/add")
def add_todo(content):
    # 在这里直接连数据库
    conn = sqlite3.connect('todo.db')
    cursor = conn.cursor()
    cursor.execute("INSERT INTO todos ...", (content,))
    conn.commit()
    # 在这里直接返回JSON
    return {"status": "ok"}
```
- API、SQL、业务逻辑混在一起
- 改数据库要改这里，改API也要改这里
- 想换数据库，整个文件重写

**✅ 模块化版本：分层清晰**

```
modules/
├── api/          # 只处理 HTTP 请求
├── services/     # 只处理业务逻辑
└── repository/   # 只处理数据库操作
```

```python
# api/todos.py  → 只处理请求响应
from services.todo_service import TodoService

@router.post("/add")
def add_todo(request: AddTodoRequest):
    service = TodoService()
    result = service.add_todo(request.content)

# services/todo_service.py → 只处理业务逻辑
from repository.todo_repo import TodoRepository

class TodoService:

```

**好处**：
- 想改数据库，只改 `repository/`，其他不动
- 想改业务逻辑，只改 `services/`
- 想换 Web 框架，只改 `api/`

### 模块化原则

1. **一个模块一个职责**：不要让它干太多活
2. **明确的接口**：告诉外面我能做什么
3. **隐藏实现细节**：内部怎么干不用告诉别人
4. **尽量少依赖别人**：改一个地方不要牵动全身

## 1.5 组件化：像搭积木一样拼项目

### 什么是组件化？

模块化是按功能分，组件化是按业务分。

> **把业务拆成独立组件，像搭积木一样拼出整个系统。**

示例：电商网站

```
电商网站
├── 用户组件          （登录、注册、个人信息）
├── 商品组件          （商品列表、搜索、详情）
├── 购物车组件        （加购、删除、修改数量）
├── 订单组件          （下单、支付、订单列表）
└── 支付组件          （微信支付、支付宝）
```

每个组件：
- 独立开发
- 独立测试
- 可以替换
- 可以复用

### 组件之间怎么通信？

**❌ 错误：组件直接互相调用**
```python
# 订单组件直接改购物车数据
class Order:
    def create(self):
        # 直接改 ShoppingCart.cart，耦合太紧密
        ShoppingCart.clear()
```

**✅ 正确：通过事件/消息通信**
```python
# 订单创建完成，发一个事件
order_created.send(order_id=order.id)

# 购物车组件监听这个事件，自己清空
@on(order_created)
def clear_cart(event):
    cart.clear(event.user_id)
```

好处：
- 订单组件不需要知道购物车的存在
- 加新功能，只要加新监听，不用改订单代码

## 1.6 抽象思维：抓住重点，忽略细节

### 什么是抽象？

**把不重要的细节藏起来，只关注你需要的。**

例子：开车
- 你不需要知道发动机原理、油路怎么走
- 你只要知道：油门走，刹车停，方向盘转弯
- 油门、刹车、方向盘就是抽象出来的接口

### 代码中的抽象

```python
# 我们想用数据库，不需要知道连接细节

from database import get_connection

# ✅ 抽象之后，用起来很简单
conn = get_connection()
cursor = conn.execute("SELECT * FROM users")

# 如果没有抽象，每次用都要写一堆：
def get_connection():
    host = config.get('DB_HOST')
    port = config.get('DB_PORT')
    user = config.get('DB_USER')
    password = config.get('DB_PASSWORD')
    # 建立连接...
    # 处理异常...
    # 返回连接...
```

### 抽象的好处

1. **减少重复代码**：连接逻辑只写一次，到处用
2. **改一处全生效**：换数据库，只改 `get_connection`
3. **用的时候不用想细节**：拿来就用

### 怎么练习抽象？

- 当你发现同样代码写了三遍 → 抽象成函数
- 当你发现多个类有相同行为 → 抽象成基类
- 当你发现细节总是干扰你 → 把细节藏到下层

## 1.7 流程控制：让代码按顺序执行

### 三种基本流程

所有程序都只有这三种：

| 结构 | 说明 |
|------|------|
| **顺序执行** | 从上到下一行一行走 |
| **条件分支** | 满足条件才走这里 |
| **循环** | 重复执行直到结束 |

就这三种，能写出所有程序！

### 流程控制最佳实践

#### 1. 少用 `else`，尽量提前返回

前面讲过了，这里再强调：

```python
# ❌ 嵌套太多
if user.is_active:
    if user.has_permission:
        do_something()
    else:
        return error("no permission")
else:
    return error("inactive")

# ✅ 提前返回
if not user.is_active:
    return error("inactive")
if not user.has_permission:
    return error("no permission")
do_something()
```

#### 2. 长条件提前放

```python
# 把更容易不满足的条件放前面，提前过滤掉
# 这样大部分情况走不到后面

# 如果大部分用户没登录，先检查登录
if not user.logged_in:
    return error

# 再检查其他条件
```

#### 3. 循环里尽量少做

```python
# ❌ 循环里做太多
for item in list:
    connect_db()
    query()
    process()
    close_db()

# ✅ 数据库连接放外面
connect_db()
for item in list:
    query()
    process()
close_db()
```

## 第一章 总结

### 核心要点回顾

| 要点 | 一句话总结 |
|------|-----------|
| **需求拆解** | 大问题拆小，直到能直接写代码 |
| **复杂度管理** | 控制函数长度，减少嵌套 |
| **面向对象** | 把数据和行为打包在一起 |
| **模块化** | 高内聚低耦合，分层清晰 |
| **组件化** | 按业务拆分，像搭积木 |
| **抽象思维** | 藏细节，留接口 |
| **流程控制** | 提前返回，减少嵌套 |

### 常见 vibecoding 陷阱

❌ 不拆解需求，上来就写
❌ 上千行都在一个文件
❌ 嵌套五层，自己都绕晕
❌ 数据和行为分开，到处传参数
❌ 所有功能混在一起，改不动

✅ 拆完再写，分层清晰
✅ 函数短小，职责单一
✅ 提前返回，扁平化
✅ 该用 OOP 就用 OOP
✅ 抽象重复代码，不复制粘贴
