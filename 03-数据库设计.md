## 3.1 数据库选型：用什么数据库？

### 两大类数据库

| 类型 | 代表 | 特点 | 什么时候用 |
|------|------|------|-----------|
| **关系型数据库** | MySQL、PostgreSQL、SQLite | 表结构固定，支持 SQL，支持事务 | ✅ 大多数业务系统 |
| **非关系型** | Redis、MongoDB | 灵活，快，结构不固定 | ✅ 缓存、日志、高并发场景 |

### 关系型 vs 非关系型对比

#### 关系型数据库（MySQL 为例）

**优点**：
- ✅ 事务支持：要么全成，要么全败，不会出错
- ✅ SQL 强大：复杂查询很方便
- ✅ 数据一致性：保证数据正确
- ✅ 成熟稳定：用了几十年了

**缺点**：
- ❌ 海量数据下，水平扩展麻烦
- ❌ 存储超大JSON 不方便

#### 非关系型（Redis 为例）

**优点**：
- ✅ 极快：内存读写，微秒级响应
- ✅ 简单：结构简单，就是 key-value

**缺点**：
- ❌ 不适合存大量持久化业务数据
- ❌ 事务支持弱
- ❌ 数据可能丢

### 📌 选型建议

| 场景 | 选什么 |
|------|--------|
| 用户数据、订单、业务数据 | **MySQL**（关系型） |
| 缓存（热点数据加速） | **Redis** |
| 日志、事件溯源 | MongoDB |
| 项目太小，不想装服务 | SQLite（文件数据库，够用） |

> **新手建议**：先从 MySQL 学起，绝大多数项目都够用。

## 3.2 SQL 基础：必须会的几个操作

### 四大基本操作（CRUD）

```sql
-- C: Create → 创建
INSERT INTO users (name, email, password)
VALUES ('张三', 'zhangsan@example.com', 'hashedpassword');

-- R: Read → 读取
SELECT * FROM users WHERE id = 1;

-- U: Update → 更新
UPDATE users SET email = 'new@example.com' WHERE id = 1;

-- D: Delete → 删除
DELETE FROM users WHERE id = 1;
```

### 条件查询

```sql
-- 等于
SELECT * FROM users WHERE id = 1;

-- 大于小于
SELECT * FROM products WHERE price > 100;

-- 模糊搜索
SELECT * FROM users WHERE name LIKE '%张%';

-- 多个条件
SELECT * FROM todos WHERE user_id = 1 AND completed = 0;

-- 排序
SELECT * FROM todos ORDER BY created_at DESC;

-- 分页
SELECT * FROM todos ORDER BY created_at DESC LIMIT 10 OFFSET 0;
```

### 联表查询

```sql
-- 查用户 1 的所有待办
-- users 表和 todos 表关联

-- 写法 1
SELECT todos.*
FROM todos
JOIN users ON todos.user_id = users.id
WHERE users.id = 1;

-- 写法 2（子查询）
SELECT *
FROM todos
WHERE user_id IN (SELECT id FROM users WHERE id = 1);
```

### 常见错误

❌ 不用 `SELECT *`，只查需要的字段
```sql
-- ❌ 不要这么写
SELECT * FROM users;

-- ✅ 只查需要的，快很多
SELECT id, name, email FROM users;
```

❌ 不要在大表上 `SELECT COUNT(*)`，很慢
- 知道就行，真需要的时候再说

## 3.3 数据表设计：从需求到表结构

### 设计步骤

**第一步：找实体**

需求：做一个待办应用，用户可以加待办，可以标记完成。

有哪些实体？
- 用户 → `users` 表
- 待办 → `todos` 表

**第二步：找关系**

- 一个用户可以有多个待办
- 一个待办只属于一个用户
→ **一对多**关系，在 `todos` 加 `user_id` 外键

**第三步：设计每个表的字段**

`users` 表：

| 字段名 | 类型 | 说明 | 约束 |
|--------|------|------|------|
| id | INT | 主键 | PRIMARY KEY AUTO_INCREMENT |
| username | VARCHAR(50) | 用户名 | NOT NULL, UNIQUE |
| email | VARCHAR(100) | 邮箱 | NOT NULL, UNIQUE |
| password_hash | VARCHAR(255) | 密码哈希 | NOT NULL |
| created_at | DATETIME | 创建时间 | DEFAULT CURRENT_TIMESTAMP |
| updated_at | DATETIME | 更新时间 | DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP |

`todos` 表：

| 字段名 | 类型 | 说明 | 约束 |
|--------|------|------|------|
| id | INT | 主键 | PRIMARY KEY AUTO_INCREMENT |
| user_id | INT | 属于哪个用户 | NOT NULL, FOREIGN KEY REFERENCES users(id) |
| content | TEXT | 内容 | NOT NULL |
| completed | BOOLEAN | 是否完成 | DEFAULT FALSE |
| created_at | DATETIME | 创建时间 | DEFAULT CURRENT_TIMESTAMP |
| updated_at | DATETIME | 更新时间 | DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP |

**第四步：写建表 SQL**

```sql
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL UNIQUE,
    email VARCHAR(100) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

CREATE TABLE todos (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    content TEXT NOT NULL,
    completed BOOLEAN DEFAULT FALSE,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
        ON DELETE CASCADE ON UPDATE CASCADE
);
```

`ON DELETE CASCADE`：删用户，用户的待办一起删，省事。

## 3.4 数据建模：三范式，帮你避坑

### 第一范式：字段不可再分

**❌ 违反第一范式**：一个字段存多个值
```
一个用户有多个电话，存在一个字段里：
phone: "13800000000,13900000000"
```
- 没法查，没法索引，没法单独改

**✅ 符合第一范式**：分开存
- 要存多个电话，单独建 `user_phones` 表

### 第二范式：消除部分依赖

简单说：**一个表只描述一个东西**。

不用记，理解就行。

### 第三范式：消除传递依赖

**❌ 违反第三范式**：
```
orders 订单表：
order_id
user_name  ← 这里不对！
user_email ← 这里不对！
...
```
- `user_name` 和 `user_email` 依赖 `user_id`，不直接依赖 `order_id`
- 这就是传递依赖

**✅ 正确做法**：
```
orders 表：
order_id
user_id ← 只存 user_id，够了

users 表：
user_id
user_name
user_email
```
- 需要用户名，JOIN 查询就能拿到
- 改用户名，只改 users 表，不用改 orders

### 三范式总结

| 范式 | 一句话 |
|------|--------|
| 第一范式 | 一个字段只存一个值 |
| 第二范式 | 一个表只干一件事 |
| 第三范式 | 不存其他表已经有的信息，只存 ID |

**好处**：
- ✅ 没有数据冗余
- ✅ 更新方便，只改一个地方
- ✅ 不会出现数据不一致

**反范式**：需要的时候，可以适当冗余一下，加快查询。比如：商品列表页要显示用户名，每次 JOIN 也麻烦，可以把 `username` 存在商品表里。但要想好更新不同步问题。

## 3.5 性能优化：索引设计

### 什么是索引？

- 像书的目录，想找哪章直接翻过去
- 没有索引，要从头到尾一页一页找（全表扫描）
- 有索引，直接定位，快很多

### 哪些字段要加索引？

✅ **必须加索引**：
- 主键（自动加了）
- 外键（用来 JOIN，一定要加）
- WHERE 条件经常用到的字段
- ORDER BY、GROUP BY 用到的字段

❌ **不要加索引**：
- 很少用到的字段
- 性别（只有男/女，区分度太低）
- 非常长的 TEXT 字段

### 索引示例

```sql
-- 我们经常这么查：
SELECT * FROM todos WHERE user_id = 1 AND completed = 0;

-- 给这两个字段加联合索引
CREATE INDEX idx_user_id_completed ON todos(user_id, completed);
```

**最左前缀原则**：
- 索引 `(a, b, c)` 能用到 `a`、`(a,b)`、`(a,b,c)` 的查询
- 不能用到 `b`、`(b,c)` 的查询

所以：把区分度高的放左边。

### Explain 看SQL 性能

```sql
EXPLAIN SELECT * FROM todos WHERE user_id = 1;
```
看 `type` 列：
- `type: ALL` → 全表扫描，慢，需要加索引
- `type: ref` → 用到索引，好

### 常见索引错误

❌ 索引建了，但是用不上
```sql
SELECT * FROM users WHERE YEAR(created_at) > 2024;
-- 用不上索引！因为字段用了函数

-- ✅ 这么写就能用上
SELECT * FROM users WHERE created_at >= '2024-01-01';
```

❌ 索引太多
- 每个索引插入更新都会变慢
- 一般一个表索引不超过 5 个

## 3.6 查询优化

### 避免大事务

```sql
-- ❌ 不要在一个事务里做太多事
START TRANSACTION;
-- 一大堆操作，很慢
COMMIT;

-- ✅ 该拆就拆
```

### 避免 SELECT *

前面说了，只查需要的字段。

### 避免深分页

```sql
-- 翻页越往后，越慢
SELECT * FROM table ORDER BY id LIMIT 10 OFFSET 100000;

-- ✅ 优化，用主键范围
SELECT * FROM table WHERE id > 100000 ORDER BY id LIMIT 10;
```

### N+1 查询问题

**❌ N+1，很慢**：
```python
# 先查 10 个用户
users = query("SELECT * FROM users LIMIT 10")

# 对每个用户，再查他的订单
for user in users:
    orders = query(f"SELECT * FROM orders WHERE user_id = {user['id']}")
```
- 1 次查用户 + 10 次查订单 = 11 次查询 → N+1

**✅ 优化，用 IN 一次查出来**：
```python
users = query("SELECT * FROM users LIMIT 10")
user_ids = [u['id'] for u in users]
orders = query(f"SELECT * FROM orders WHERE user_id IN ({','.join(user_ids)})")
# 一共只需要 2 次查询
```

或者用 JOIN 一次查出来。

## 第三章 总结

### 核心要点

| 要点 | 一句话总结 |
|------|-----------|
| **数据库选型** | 业务数据用 MySQL，缓存用 Redis |
| **SQL 基础** | CRUD、条件查询、联表查询必须会 |
| **设计步骤** → 找实体 → 找关系 → 设计字段 → 写建表SQL |
| **三范式** | 不冗余，一个表一件事，不存多余信息 |
| **索引设计** → 外键、常用查询字段要加索引，遵循最左前缀 |
| **查询优化** | 避免N+1，避免深分页，避免SELECT * |

### 避开 vibecoding 陷阱

❌ 上来就用 MongoDB，其实业务数据需要事务
❌ 一张表几十个字段，什么都往里塞
❌ 从来不建索引，全表扫描慢到死
❌ 三范式不听，到处冗余数据，更新到处改
❌ N+1 查询不知道，性能越来越差

✅ 选对数据库，业务数据关系型
✅ 一张表一个实体，设计清晰
✅ 常用查询字段加索引，快很多
✅ 遵循三范式，减少冗余
✅ 知道 N+1 问题，一次查出需要的数据
