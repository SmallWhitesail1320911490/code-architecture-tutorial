## 5.1 Git 版本控制：代码正确地管理

### 基本工作流

```bash
# 1. 拉最新代码
git pull origin main

# 2. 创建功能分支
git checkout -b feature/add-user-login

# 3. 写代码，提交
git add .
git commit -m "feat: 实现用户登录功能"

# 4. 推到远程
git push origin feature/add-user-login

# 5. 提 Pull Request / Merge Request
# 6. 代码review，合并到 main
```

### 提交信息规范（Conventional Commits）

格式：
```
<type>: <description>

<body>

<footer>
```

type 类型：

| type | 意思 |
|------|------|
| feat | 新功能 |
| fix | 修 bug |
| docs | 改文档 |
| style | 格式调整，不影响代码 |
| refactor | 重构，不改功能 |
| perf | 性能优化 |
| test | 加测试 |
| chore | 改构建/工具 |

例子：
```
feat: 添加用户登录接口

- 实现密码验证
- 实现 JWT Token 生成
```

好处：
- ✅ 一目了然，知道这个提交干了啥
- ✅ 可以自动生成 changelog
- ✅ 别人看历史能看懂

### 分支规范

**推荐：GitHub Flow**（简单好用）

- `main` 分支永远是可部署的
- 新功能：从 main 拉 `feature/xxx`
- 做完了 PR 合并回 main
- 修复 bug：从 main 拉 `fix/xxx`，做完合并

**如果需要发版：**可以加 GitFlow，但对中小项目没必要，GitHub Flow 足够简单。

### 常见 Git 错误

❌ 直接在 main 分支改代码
- 线上出问题没法拉新代码，只能硬合

✅ 永远分支开发，合并完再测

❌ 一个提交几百行，改了五个功能
- 出了问题找不到哪次提交引入的 bug

✅ 一个功能一个提交，一个 bug 一个修复，拆小一点

❌ 提交信息写 "fix bug" "update" 什么都不说
- 过了三个月自己都不知道改了啥

✅ 写清楚改了什么，为什么改

## 5.2 模块化开发

前面第一章讲过，这里再强调工程上的实践：

### 按功能分包 / 分文件夹

```
my_project/
├── api/              # API 接口
│   ├── __init__.py
│   ├── dependencies.py
│   └── endpoints/
│       ├── auth.py
│       ├── todos.py
│       └── users.py
├── services/         # 业务逻辑
│   ├── __init__.py
│   ├── auth_service.py
│   └── todo_service.py
├── models/           # 数据模型
│   ├── __init__.py
│   ├── schemas.py
│   └── repositories.py
├── core/             # 核心工具
│   ├── __init__.py
│   ├── config.py
│   ├── security.py
│   └── logger.py
└── main.py           # 入口
```

### 循环依赖怎么办？

```python
# a.py 导入 b
# b.py 导入 a
→ 循环依赖，运行报错
```

解决方法：
1. **抽公共代码到 c.py**，a 和 b 都导 c
2. **把导入放函数里面**，不要在文件顶部导
3. **重构模块**，说明拆分不合理，合起来或者重新拆

### 导入规范

```python
# 顺序：标准库 → 第三方 → 本地
import os
import sys

import bcrypt
import fastapi

from my_project.core.config import settings
from my_project.models.schemas import User
```

干净整齐，一目了然。

## 5.3 日志系统：出了问题能找到原因

### 为什么要日志？

- 线上出了问题，你不能断点调试
- 靠日志看发生了什么
- 用户说"支付失败"，你看日志就知道哪一步错了

### 正确使用日志

**❌ 不要只用 print**
```python
print("这里到了")
print(f"user_id={user_id}")
```
- 打印到控制台，线上没了
- 没有级别，没有时间，找不到

**✅ 用标准日志库**

```python
import logging

logger = logging.getLogger(__name__)

@app.post("/login")
def login(username: str, password: str):
    logger.info(f"尝试登录: username={username}")
    try:
        user = auth_service.login(username, password)
        logger.info(f"登录成功: user_id={user.id}")
        return user
    except Exception as e:
        logger.warning(f"登录失败: username={username}, error={str(e)}")
        raise
```

### 日志级别

| 级别 | 什么时候用 |
|------|-----------|
| DEBUG | 开发调试信息，线上默认关闭 |
| INFO | 正常流程，比如"用户登录成功" |
| WARNING | 不影响正常运行，但是要注意 |
| ERROR | 出错误了，需要处理 |
| CRITICAL | 严重错误，系统挂了 |

一般线上开 INFO 及以上，DEBUG 开了日志太多。

### 日志实践

1. **要记什么**：
   - 请求来了，记一下请求 ID、用户 ID
   - 出错了，把错误堆栈记下来
   - 关键操作（支付、删数据）记日志

2. **不要记什么**：
   - 不要记用户密码
   - 不要记敏感信息（Token、身份证）
   - 不要记超大响应体

3. **日志格式要对**：
   ```
   时间 级别 模块 消息
   2024-03-31 10:00:00 INFO auth.login 尝试登录: username=zhangsan
   ```
   方便排查问题。

## 5.4 错误监控：早点知道线上错了

### 为什么需要监控？

- 用户遇到错了不告诉你
- 你得自己知道线上出错了
- 出问题了能收到报警

### 简单方案：邮件报警

把错误日志发邮箱：
```python
import logging
from logging.handlers import SMTPHandler

# 配置错误发邮件
# ERROR 级别以上发邮件
mail_handler = SMTPHandler(...)
mail_handler.setLevel(logging.ERROR)
logger.addHandler(mail_handler)
```
- 简单，不用第三方服务
- 错了马上收到邮件

### 专业方案：Sentry

[Sentey](https://sentry.io/) 免费够用：
- 自动收集错误
- 有堆栈信息，哪一行错了一目了然
- 报警通知（邮件、Slack、企业微信）
- 统计错误频率

接入超简单：
```python
import sentry_sdk
from sentry_sdk.integrations.fastapi import FastApiIntegration

sentry_sdk.init(
    dsn="YOUR_DSN",
    traces_sample_rate=1.0
)
```

出了错，你手机马上收到通知。

### 错误处理原则

1. **不要吞错误**
```python
try:
    do_something()
except:
    pass  # ❌ 错了不说，出问题找不到
```

正确：
```python
try:
    do_something()
except Exception as e:
    logger.error("操作失败", exc_info=True)  # ✅ 记日志
    raise  # ✅ 继续抛，让上层处理或者返回错误
```

2. **友好的错误信息**
- 给用户："操作失败，请稍后重试"
- 给日志：详细错误信息+堆栈

## 5.5 Docker 容器化部署：一次构建，到处运行

### 什么是 Docker？

- 把你的应用 + 依赖 + 环境 → 打包成一个"镜像"
- 任何机器跑 Docker，直接启动你的镜像就能用
- 不用管本地是什么环境，不会出现"我这里能跑啊"问题

### 最简单的 Dockerfile（Python FastAPI 例子）

```dockerfile
# 基础镜像
FROM python:3.11-slim

# 设置工作目录
WORKDIR /app

# 复制依赖文件
COPY requirements.txt .

# 安装依赖
RUN pip install --no-cache-dir -r requirements.txt

# 复制代码
COPY . .

# 暴露端口
EXPOSE 8000

# 启动命令
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

就这几行，搞定。

### docker-compose.yml 本地开发

```yaml
version: '3.8'
services:
  web:
    build: .
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=mysql://db:3306/mydb
    depends_on:
      - db

  db:
    image: mysql:8.0
    environment:
      - MYSQL_ROOT_PASSWORD=password
      - MYSQL_DATABASE=mydb
    volumes:
      - mysql_data:/var/lib/mysql

  redis:
    image: redis:7-alpine

volumes:
  mysql_data:
```

本地跑：
```bash
docker-compose up -d
```
自动帮你起 web、mysql、redis，不用一个个装。

### 部署流程

```bash
# 1. 本地或服务器构建镜像
docker build -t my-app:latest .

# 2. 跑起来
docker run -d -p 8000:8000 --env-file .env my-app:latest
```

### Docker 好处

- ✅ 环境一致：开发测试生产环境一样
- ✅ 部署简单：拉镜像启动就完了
- ✅ 版本控制：镜像打标签，回滚方便
- ✅ 隔离：各应用互不影响

## 5.6 环境配置：开发/生产分开

### 为什么分开？

- 开发：用开发数据库，开 debug 日志
- 生产：用生产数据库，关 debug，性能优先
- 不能把开发配置带到线上

### 正确做法：环境变量

用 `.env` 文件，放项目根目录，**不要提交到 Git**！

```
# .env 文件
APP_ENV=development
DEBUG=true
DATABASE_URL=mysql://root:password@localhost:3306/dev_db
JWT_SECRET=your_dev_secret_here
```

Python 用 pydantic 读：

```python
# core/config.py
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    app_env: str = "development"
    debug: bool = False
    database_url: str
    jwt_secret: str

    class Config:
        env_file = ".env"

settings = Settings()
```

用的时候：
```python
from core.config import settings

print(settings.debug)  # 自动从环境变量读
```

### .gitignore

一定要把这些加进 `.gitignore`：
```
.env
.env.*
*.log
__pycache__/
*.pyc
venv/
.idea/
.vscode/
```
- 不要把本地环境配置提交到 Git
- 不要把虚拟环境提交上去

## 第五章 总结

### 核心要点

| 要点 | 一句话总结 |
|------|-----------|
| **Git 规范** | 分支开发，提交信息规范，不乱改 main |
| **模块化** | 按功能分包，每个文件一个职责 |
| **日志系统** | 合理打日志，出错能排查，不记敏感信息 |
| **错误监控** | 线上错了自动报警，早点知道早点修 |
| **Docker 部署** | 容器化，一次构建到处跑，环境一致 |
| **环境分离** | 开发生产配置分开，用环境变量，不提交 .env |

### 避开 vibecoding 陷阱

❌ 直接在 main 分支写代码，提交写"update"
❌ 所有代码堆一块，不拆分模块
❌ 出了问题线上没有日志，两眼一抹黑
❌ 错了吞了不记日志，用户不说你不知道
❌ 把密码密钥提交到 Git，全世界都看见了
❌ 开发环境写死在代码，上线要改半天

✅ 分支开发，提交信息清晰
✅ 模块化分文件夹，职责清晰
✅ 合理打日志，出错能排查
✅ 错误监控，自动报警
✅ Docker 打包，部署简单
✅ 环境变量管理配置，不提交敏感信息
