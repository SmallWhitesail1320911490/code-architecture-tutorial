## 6.1 依赖管理：管好你的第三方库

### 什么是依赖？

你写代码不用什么都从头写，用别人写好的：
```python
import fastapi
import bcrypt
import pydantic
```
这些就是**依赖**。

### 依赖锁定很重要

**❌ 错误做法：不锁定版本**
```txt
# requirements.txt
fastapi
bcrypt
pydantic
```
- 今天装是 0.100，明天装变成 0.101，可能 break 你的代码
- 你本地能跑，服务器新装就炸了

**✅ 正确做法：锁定版本**

分两种文件：

```txt
# requirements.in  ← 你写这里
fastapi>=0.100.0
bcrypt>=4.0.0
pydantic>=2.0.0
```

```txt
# requirements.txt  ← 自动生成，锁定具体版本
fastapi==0.100.0
bcrypt==4.0.1
pydantic==2.5.3
starlette==0.27.0
... 所有依赖都锁定版本 ...
```

### Python 依赖管理工具选择

| 工具 | 特点 | 推荐 |
|------|------|------|
| **pip + requirements.txt** | 简单直接，够用 | ✅ 新手推荐 |
| **poetry** | 现代，自动锁定，更好的依赖解析 | ✅ 新项目推荐 |
| **pipenv** | 整合 pip 和 virtualenv | 可以用，现在 poetry 更流行 |

### poetry 示例

```bash
# 初始化
poetry init

# 加依赖
poetry add fastapi bcrypt pydantic

# 安装
poetry install

# poetry.lock 会锁定所有版本
# 把 poetry.lock 提交到 Git
# 别人拉下来 poetry install 就和你一模一样版本
```

### 依赖原则

1. **不要乱加依赖**：能用标准库就不用第三方
   - `json` 标准库够用，不用再装第三方 json 库
   - 确实需要再加

2. **定期更新依赖**：
   - 安全漏洞会被发现，更新修复
   - 一次更一点，别一下更一百个，炸了找不到谁干的

3. **删掉不用的依赖**：
   - 不用了就删掉，减少体积，减少安全风险

### 常见依赖错误

❌ 把 `venv/` 提交到 Git
- 每个人环境不一样，体积又大
- ✅ `.gitignore` 加上 `venv/`

❌ 依赖版本范围太松
- `*` 随便更，哪天就炸了
- ✅ 锁定大版本，`fastapi>=0.100,<0.101`

❌ 开发依赖和生产依赖混在一起
- 测试工具什么的，生产不用装
- ✅ poetry 分 `--dev`，requirements 分开发生产

## 6.2 环境管理：开发环境不乱

### 为什么要虚拟环境？

- 不同项目依赖版本不一样
- 项目 A 需要 Django 3，项目 B 需要 Django 4
- 虚拟环境隔开，互不影响

### conda vs venv vs poetry

| 工具 | 用途 | 适用场景 |
|------|------|-----------|
| **venv** | Python 虚拟环境 | 纯 Python 项目，够用 |
| **conda** | 不仅 Python，还能装 C/C++ 库 | 数据科学、机器学习项目 ✅ |
| **poetry | 依赖+虚拟环境一起管理 | 现代 Python 项目 ✅ |

### 新手推荐使用流程

#### 如果是纯 Python Web 项目 → poetry

```bash
# 进项目目录
cd my-project

# 创建虚拟环境
poetry env use python3.11

# 激活
poetry shell

# 装依赖
poetry install

# 跑你的项目
uvicorn app.main:app --reload
```

#### 如果是机器学习项目 → conda

```bash
# 创建环境，指定 Python 版本
conda create -n my-project python=3.10

# 激活
conda activate my-project

# 装依赖
conda install pytorch torchvision
pip install -r requirements.txt
```

### 最佳实践

1. **每个项目一个环境**
   - 不要都装在系统 Python 里
   - 时间长了，什么版本都有，乱成一锅粥

2. **不要用 sudo pip install**
   - 会搞坏系统 Python
   - 用虚拟环境，用户权限就够了

3. **记录环境信息**
   - `requirements.txt` 或 `poetry.lock` 提交 Git
   - 别人拿到，一键建好环境

4. **开发环境** vs **生产环境**

| 开发 | 生产 |
|------|------|
| 有调试工具 | 不要调试工具 |
| 开 debug | 关 debug |
| 用开发数据库 | 用生产数据库 |
| 热重载开着 | 热重载关掉 |

## 6.3 Docker 部署（生产上线）

前面第五章讲了 Docker，这里讲生产部署：

### 生产 Dockerfile 优化

```dockerfile
# 多阶段构建，减小镜像体积

# 第一阶段：构建
FROM python:3.11-slim as builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user -r requirements.txt

# 第二阶段：最终镜像
FROM python:3.11-slim
WORKDIR /app
# 只复制装好的依赖，比从头装小很多
COPY --from=builder /root/.local/lib/python3.11/site-packages /usr/local/lib/python3.11/site-packages
COPY . .

EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### 生产 docker-compose

```yaml
version: '3.8'
services:
  web:
    image: my-app:latest
    restart: always  # ✅ 挂了自动重启
    ports:
      - "127.0.0.1:8000:8000"  # ✅ 只绑本地，不要暴露直接外网
    environment:
      - APP_ENV=production
      - DEBUG=false
    env_file:
      - .env.prod
    depends_on:
      - db
      - redis

  db:
    image: mysql:8.0
    restart: always
    volumes:
      - mysql_data:/var/lib/mysql
    environment:
      MYSQL_ROOT_PASSWORD: ${DB_PASSWORD}

  redis:
    image: redis:7-alpine
    restart: always
    volumes:
      - redis_data:/data

  nginx:
    image: nginx:alpine
    restart: always
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
      - ./cert:/etc/nginx/cert
    depends_on:
      - web

volumes:
  mysql_data:
  redis_data:
```

- `restart: always` → 容器挂了自动拉起来
- Nginx 反向代理 → HTTPS  termination
- 数据卷 → 数据存在本地，容器删了数据还在

### 部署流程

```bash
# 1. 拉最新代码
git pull origin main

# 2. 构建镜像
docker build -t my-app:latest .

# 3. 重启服务
docker-compose up -d --force-recreate web

# 4. 看日志
docker-compose logs -f web
```

简单，可靠，一键完成。

### 环境变量管理

生产环境的 `.env.prod` 不要提交到 Git！
```
# .env.prod
DB_PASSWORD=your_strong_prod_password
JWT_SECRET=your_strong_jwt_secret
```
- 只存在生产服务器上
- 本地开发用 `.env`

## 6.4 项目文档

项目要文档，不然过三个月你自己都忘了。

### 至少要有 README.md

```markdown
# 项目名称

一句话介绍项目。

## 环境要求

- Python >= 3.10
- MySQL >= 8.0
- Redis >= 6

## 本地开发

```bash
git clone xxx
cd xxx
poetry install
poetry shell
cp .env.example .env
# 改 .env
docker-compose up -d db redis
python -m alembic upgrade head
uvicorn app.main:app --reload
```

## 项目结构

```
项目结构...
```

## API 文档

- http://localhost:8000/docs

## 部署

... 部署步骤 ...
```

README 写清楚，新来的人（或者三个月后的你）照着就能跑起来。

### 变更记录

CHANGELOG.md 记一下每个版本改了什么：
```markdown
# 变更记录

## v1.1.0 - 2024-03-31
- 新增：用户权限控制
- 修复：登录 Token 过期问题

## v1.0.0 - 2024-03-01
- 首次发布
```

## 6.5 常见问题处理

### "它在我本地能跑啊，线上为什么不行？"

都是环境不一致导致的：
- ✅ 用 Docker，环境一致就没这问题
- ✅ 锁定依赖版本，大家都装一样的
- ✅ 检查配置，线上配置对不对

### "我更新了依赖，好多东西坏了"

- ✅ 一次更一个依赖，更完测试
- ✅ 用 lock 文件，生产和开发版本一致
- ✅ 锁定大版本，别乱更大版本

### "虚拟环境搞乱了"

```bash
# 删了重建，很简单
rm -rf venv
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```
坏了就重建，不要折腾。

## 如何避免 vibecoding？

记住这个检查清单，写代码前过一遍：

- [ ] 需求拆解了吗？大问题拆小了吗？
- [ ] 目录结构分好模块了吗？
- [ ] 分层对吗？API / Service / Repository 分开了吗？
- [ ] 数据库表设计符合三范式吗？索引加了吗？
- [ ] 密码加密了吗？HTTPS 用了吗？
- [ ] 权限判断做了吗？
- [ ] 日志会打吗？错了能查到吗？
- [ ] 依赖版本锁定了吗？
- [ ] 环境配置分开开发生产了吗？

**每次做对一件事，日积月累，就不会 vibecoding 了。**
